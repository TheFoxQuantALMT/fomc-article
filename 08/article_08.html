The Most Interesting Part of a Simulation Is Not the Prediction

A stateless statistical engine that tells you why two models disagree — Pearson correlation, Welch's t-test, Cohen's d, CRPS, Diebold-Mariano, and the case for effect size over p-values.

I run two completely independent models of the Federal Reserve's interest rate decisions. One is a Monte Carlo simulation with Bayesian belief updating. The other is a multi-agent LLM deliberation where AI personas argue their way to a vote. They both predict the same thing: the Federal Funds Rate.

The prediction itself is fine. But the interesting part — the part that actually tells you something — is when they disagree.

That is why the comparison service exists. It is Phase 3 of the pipeline, 1,375 lines of Python, and its entire job is to take outputs from two modeling tracks, lay them side by side, and answer: how different are these results, and should we care?

Three Metrics, Three Stories

def compare_tracks(mc_mean_rate, mc_std_rate, mc_beliefs,
                   llm_rate, llm_votes):
    rate_diff = abs(mc_mean_rate - llm_rate)

    # Per-agent belief correlation
    common = set(mc_beliefs.keys()) & set(llm_votes.keys())
    if len(common) >= 3:
        mc_vals = [mc_beliefs[a] for a in common]
        llm_vals = [llm_votes[a] for a in common]
        corr, _ = stats.pearsonr(mc_vals, llm_vals)

    if rate_diff < 0.1 and abs(corr) > 0.7:
        agreement = "strong"
    elif rate_diff < 0.25 and abs(corr) > 0.4:
        agreement = "moderate"
    else:
        agreement = "weak"

The rate difference tells you how far apart the final numbers are. The dissent divergence tells you whether internal dynamics played out similarly. And the Pearson correlation on per-agent beliefs tells you whether the same agents ended up in the same place — which is the most revealing signal. If Agent A (a known hawk) ended up dovish in the MC track but hawkish in the LLM track, that is a qualitative disagreement worth investigating, not just a numerical one.

"Strong" agreement requires both a rate gap under 10bp and per-agent correlation above 0.7. In practice, most meetings land in "moderate" — which is exactly what you would expect from two fundamentally different approaches arriving at similar conclusions through different reasoning.

Treatment Effects: What Variables Actually Change Outcomes

The service answers an experimental question: if I change one input, does the output change? Run the MC simulation twice — baseline and with an oil price shock — and feed both distributions to the treatment endpoint:

def calculate_treatment_effect(baseline_rates, treatment_rates, treatment_name):
    b = np.array(baseline_rates)
    t = np.array(treatment_rates)

    pooled_std = np.sqrt((np.std(b, ddof=1)**2 + np.std(t, ddof=1)**2) / 2)
    effect_size = float((np.mean(t) - np.mean(b)) / pooled_std)  # Cohen's d

    _, p_value = stats.ttest_ind(b, t, equal_var=False)  # Welch's t-test

    # Bootstrap 95% CI on the mean difference
    rng = np.random.default_rng(42)
    diffs = []
    for _ in range(10_000):
        bs = rng.choice(b, size=len(b), replace=True)
        ts = rng.choice(t, size=len(t), replace=True)
        diffs.append(float(np.mean(ts) - np.mean(bs)))
    ci = [float(np.percentile(diffs, 2.5)), float(np.percentile(diffs, 97.5))]

Three layers of rigor. First, Welch's t-test (unequal variance — important when treatment might change the spread). Second, Cohen's d effect size, because p-values are notoriously misleading with large simulation sample sizes — a p-value of 0.001 means nothing if the effect size is 0.02. Third, a 10,000-iteration bootstrap for the confidence interval, because the bootstrap makes no distributional assumptions about the rate distribution (which is often multimodal since rates cluster at 25bp increments).

The fixed seed is deliberate. Reproducibility is non-negotiable in a pipeline that might be audited.

Proper Scoring Rules: Are You Actually Good at This?

Producing a forecast is easy. Knowing whether your forecasts are well-calibrated is hard. The scoring module implements three proper scoring rules:

CRPS (Continuous Ranked Probability Score) evaluates the entire forecast distribution against the single observed outcome:

def crps(forecast_samples, observation):
    x = np.sort(np.asarray(forecast_samples).ravel())
    n = len(x)
    term1 = float(np.mean(np.abs(x - observation)))
    # O(n log n) sorted-array identity for pairwise differences
    indices = np.arange(n)
    term2 = float(np.sum((2 * indices - n + 1) * x) / (n * n))
    return term1 - abs(term2)

Unlike point-estimate metrics (MAE, RMSE), CRPS rewards a tight, well-centered distribution and penalizes overconfidence or bias.

Log score estimates density at the observation using Gaussian KDE, then returns the negative log density. This is the most punishing metric — it goes to infinity if your forecast assigns zero probability to what actually happened. There is no hiding from the log score.

Brier score evaluates binary outcomes: did your predicted probability of a rate hike match what happened?

The Diebold-Mariano Test

The question every dual-model system needs: is one model statistically better than the other?

def diebold_mariano(errors_1, errors_2, loss="squared"):
    e1, e2 = np.asarray(errors_1), np.asarray(errors_2)
    n = min(len(e1), len(e2))

    if loss == "squared":
        d = e1[:n]**2 - e2[:n]**2
    else:
        d = np.abs(e1[:n]) - np.abs(e2[:n])

    d_bar = float(np.mean(d))
    d_var = float(np.var(d, ddof=1))
    dm_stat = d_bar / math.sqrt(d_var / n)
    p_value = 2.0 * (1.0 - stats.t.cdf(abs(dm_stat), df=n - 1))

The Diebold-Mariano test answers the question every dual-model system needs to confront: is one model actually better at forecasting, or has it just been luckier over a particular stretch? The test works by computing the difference in loss (squared error or absolute error) between two models at each time step, then testing whether the mean difference is statistically distinguishable from zero. The basic implementation uses standard variance estimation of the loss differences. A more robust version would use HAC (Heteroscedasticity and Autocorrelation Consistent) variance estimation via the Newey-West method to account for serial correlation — important because forecast errors in consecutive FOMC meetings are not independent. With only 40+ meetings, the practical difference is small, but this is a known limitation.

In the walk-forward backtest, the comparison service applies the DM test at each step as the window expands: MC errors versus LLM errors, and combined ensemble errors versus MC-only errors. Early in the backtest, the test has little power -- a handful of meetings cannot distinguish skill from luck. But as the window grows past 15-20 meetings, the test begins to deliver definitive statistical answers: does the ensemble actually outperform its components, or is the combination no better than the best individual track?

The Beauty of Stateless Design

This entire service has no database. No persistence layer. No Redis cache. No connection pools. Every request arrives with all the data it needs, gets processed through pure functions, and returns a result.

This was deliberate. The comparison service sits at Phase 3, after MC and LLM have already run. It does not need to remember anything between requests. The FastAPI layer is intentionally thin — extract data from the Pydantic model, validate minimums, call the pure analysis function, return the typed response. There is no ORM, no migration, no connection retry logic. The complexity lives in the math, not the infrastructure.

The result: it can be horizontally scaled by running more containers, restarted with zero data loss, and tested with nothing more than pytest and fabricated inputs.

Lessons Learned

Stateless services are underrated. Resisting the "give it a database" impulse made this the most reliable, testable, and deployable service in the platform.

Effect size matters more than p-values. With 10,000 MC iterations and near-zero sampling noise, almost everything is "statistically significant." Cohen's d and bootstrap CIs tell you whether the difference is meaningful, not just detectable.

Proper scoring rules keep you honest. If you produce probabilistic forecasts and only measure point accuracy, you are leaving the most important information on the table.

The DM test answers the question that matters. "Is Model A better than Model B?" requires controlling for time-series structure. The Diebold-Mariano test does this.

The comparison service is the smallest service in the platform. It is also, arguably, the one that makes everything else worth building.

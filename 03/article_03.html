When Vector Search Isn't Enough: GraphRAG for Financial Intelligence

Combining Qdrant vector search with a knowledge graph to answer questions that similarity alone cannot — structure-aware chunking, entity extraction without LLMs, and Reciprocal Rank Fusion.

Here is a question that breaks naive RAG: "What did the San Francisco Fed president say about inflation expectations after the September Beige Book was released?"

A standard vector search will return chunks that are semantically similar — paragraphs mentioning inflation expectations, San Francisco Fed passages, September Beige Book excerpts. But it cannot connect them. It does not know that Mary Daly is the San Francisco Fed president, that she gave a speech on October 3 referencing the September Beige Book, or that the Beige Book's 12th District section reported accelerating service-sector prices that she later cited.

This is the gap between retrieval and reasoning. Vector search finds similar text. What I needed was a system that understands that Speaker A represents District B, spoke about Topic C, which is measured by Indicator D — and can traverse those connections to assemble context that no single chunk contains.

The economy service in my FOMC simulation platform uses a dual-retrieval architecture: Qdrant for vector search and a Neo4j knowledge graph for entity-relationship traversal, merged using Reciprocal Rank Fusion. Here is how it works.

Why Pure Vector Search Fails for Financial Documents

Financial documents have a specific structure that resists flat embedding. Consider the FOMC minutes. They are organized into sections: "Staff Review of the Economic Situation", "Participants' Views on Current Conditions", "Committee Policy Action". Each section serves a different function — the staff review is descriptive, the participants' views are argumentative, the policy action section is prescriptive.

When you embed these as flat chunks, you lose the structural context. A chunk from the staff review that says "the unemployment rate edged down to 3.7 percent" and a chunk from participants' views that says "several participants noted the labor market remained tight" will both score well against a query about employment. But only the second one tells you what FOMC members actually thought about it — and knowing which members said it, and whether they were hawks or doves, changes the interpretation entirely.

Vector search also cannot perform multi-hop reasoning. If you ask "which indicators concern the districts that reported slowdowns," you need to: (1) find Beige Book chunks mentioning slowdowns, (2) identify which districts those came from, (3) find what indicators those districts track, and (4) retrieve recent values. That is a graph traversal, not a similarity search.

The Dual-Retrieval Architecture

A query flows through the system like this:

  1. User (or agent) asks a question with an as_of_date
  2. The query goes to both Qdrant (vector search) and the knowledge graph (entity traversal)
  3. Qdrant returns semantically similar chunks, date-bounded to as_of_date
  4. The knowledge graph expands those results by 1-2 hops to find related chunks
  5. Reciprocal Rank Fusion merges both result sets into a single ranked list
  6. MMR diversification removes near-duplicate results
  7. Optional reranking via cross-encoder

Chunking Strategy: Structure-Aware, Not Just Size-Based

The chunker does not blindly split documents every 500 tokens. It understands the structure of each document type:

Beige Books are chunked by district section. Each of the 12 districts gets its own chunks with district metadata attached:

def chunk_beige_book(beige_book_id, release_date,
                     national_summary, district_sections,
                     chunk_size=500):
    chunks = []
    # National summary as its own chunk(s)
    if national_summary:
        for i, piece in enumerate(_split_text(national_summary, chunk_size)):
            prefixed = f"[Beige Book National Summary] {piece}"
            chunks.append(DocumentChunk(
                chunk_id=f"beige_book_{beige_book_id}_national_{i}",
                text=prefixed,
                doc_type="beige_book",
                date=release_date,
                metadata={"district": "national"},
            ))
    # District sections with district metadata
    for ds in district_sections:
        district_id = ds.get("district_id", "unknown")
        for i, piece in enumerate(_split_text(ds["section_text"], chunk_size)):
            prefixed = f"[{_district_label(district_id)} District] {piece}"
            chunks.append(DocumentChunk(..., metadata={"district": district_id}))
    return chunks

FOMC minutes get structured section detection. The chunker looks for known heading patterns and splits accordingly:

_FOMC_SECTION_PATTERNS = [
    ("economic_review",    re.compile(r"staff\s+review\s+of\s+the\s+economic", re.I)),
    ("participants_views", re.compile(r"participants['\u2019]?\s+views?", re.I)),
    ("policy_action",      re.compile(r"committee\s+policy\s+action", re.I)),
    ("voting",             re.compile(r"(?:unanimous\s+vote|voting\s+for)", re.I)),
]

Speeches get speaker attribution in their chunk prefix: [Speech by Jerome Powell: Monetary Policy and the Dual Mandate]. The embedding itself carries speaker identity, making vector search speaker-aware without requiring a metadata filter.

Text splitting uses a three-tier approach: paragraph-level (keeps topics together), sentence-level (for oversized paragraphs), word-level (fallback for giant sentences). The 50-word overlap ensures continuity across chunk boundaries.

Entity Extraction: Fast, Deterministic, No LLM Required

A common GraphRAG approach calls an LLM for entity extraction on every chunk. That is expensive and non-deterministic. My entity extractor uses pattern matching instead:

_INDICATOR_PATTERNS = {
    "CPI":          re.compile(r"\bCPI\b|consumer\s+price\s+index", re.I),
    "PCE":          re.compile(r"\bPCE\b|personal\s+consumption", re.I),
    "unemployment": re.compile(r"unemployment\s+rate|jobless\s+rate", re.I),
    "GDP":          re.compile(r"\bGDP\b|gross\s+domestic\s+product", re.I),
    "payrolls":     re.compile(r"nonfarm\s+payrolls?|payroll\s+employment", re.I),
    # ... 15+ more
}

_SPEAKER_REGISTRY = {
    "powell": "Jerome Powell",
    "yellen": "Janet Yellen",
    "waller": "Christopher Waller",
    # ... all FOMC members
}

For a domain like monetary policy, the entity vocabulary is finite and well-defined. There are ~50 economic indicators, 12 districts, and roughly 30 current and recent FOMC members. Regex matching is faster, cheaper, and perfectly reproducible — every run produces the same entity set, unlike LLM-based extraction where you get different entities depending on temperature and prompt.

The extracted entities become nodes in the knowledge graph, connected to the chunks they appear in.

The Knowledge Graph

The graph is backed by NetworkX for fast in-process traversal, with Neo4j available as an optional persistence and scaling layer:

Entity types: speaker, district, indicator, topic, document
Edge types:   mentions, represents, measures, published_in, spoke_about

When a chunk from a Jerome Powell speech mentions CPI and the San Francisco district, the graph creates:

chunk:speech_powell_2024_03 --mentions--> indicator:CPI
chunk:speech_powell_2024_03 --mentions--> district:san_francisco
speaker:Jerome Powell --authored--> chunk:speech_powell_2024_03

The power is in the graph's traversal capabilities. With max_hops=2, a query can discover that: a speech chunk mentions Jerome Powell, Jerome Powell connects to FOMC minutes where he participated, those minutes discuss unemployment, and a Beige Book chunk also references unemployment. Two hops, four document types, connected by shared entities.

def expand_from_chunks(self, seed_chunk_ids, max_hops=2, max_expansion=10):
    related_chunks = set()
    frontier = {f"chunk:{cid}" for cid in seed_chunk_ids}

    for hop in range(max_hops):
        entity_nodes = set()
        for chunk_node in frontier:
            for neighbor in self._graph.predecessors(chunk_node):
                if not neighbor.startswith("chunk:"):
                    entity_nodes.add(neighbor)

        next_frontier = set()
        for ent_node in entity_nodes:
            for neighbor in self._graph.successors(ent_node):
                if neighbor.startswith("chunk:") and neighbor not in visited_chunks:
                    related_chunks.add(neighbor)
                    if len(related_chunks) >= max_expansion:
                        break
        frontier = next_frontier

    return [c.removeprefix("chunk:") for c in related_chunks]

The max_expansion=10 cap prevents the graph from returning hundreds of tangentially related chunks. In practice, 2 hops with 10 expansion chunks is the sweet spot — it surfaces genuinely relevant context without drowning the LLM in noise.

Reciprocal Rank Fusion: Merging Two Retrieval Worlds

RRF merges the vector results and graph-expanded results into a single ranked list:

def rrf_merge(qdrant_results, neo4j_results,
              alpha=0.5, k=60, top_k=15):
    """score(d) = alpha/(k + rank_qdrant) + (1-alpha)/(k + rank_neo4j)"""

    for cid, entry in entries.items():
        rank_q = qdrant_ranks.get(cid, qdrant_penalty)
        rank_n = neo4j_ranks.get(cid, neo4j_penalty)
        rrf_score = alpha / (k + rank_q) + (1 - alpha) / (k + rank_n)
        entry["rrf_score"] = rrf_score

The alpha=0.5 gives equal weight to vector and graph results. The k=60 smoothing constant (from the original Cormack et al. 2009 paper) prevents any single high-ranked item from dominating.

The penalty rank is subtle but important: items missing from one source are penalized with len(results) + 1. This ensures that a chunk appearing in both sources always outscores a chunk appearing in only one, all else being equal. A passage that is semantically similar (Qdrant) and structurally connected (graph) to the query is more likely relevant than one that only matches on one dimension.

Date-Bounded Search: The Point-in-Time Constraint

Every search respects the same point-in-time constraint as the structured data from the economy service. Dates are stored as integers (20260128 for 2026-01-28) for efficient numeric range filtering in Qdrant:

def search_date_bounded(self, query, as_of_date, top_k=15):
    date_int = self._date_str_to_int(as_of_date)
    must_conditions = [
        FieldCondition(key="date", range=Range(gt=0, lte=date_int)),
    ]
    response = self._client.query_points(
        collection_name=self._collection_name,
        query=query_embedding,
        query_filter=query_filter,
        limit=top_k,
    )

When simulating the January 2026 FOMC meeting, this filter ensures we never retrieve a Beige Book released in April, a speech given in May, or minutes published in June.

Embeddings: Ollama, Not OpenAI

Every embedding is generated by Ollama running locally using qwen3-embedding:0.6b. No external API dependency for vector operations — no OpenAI, no Cohere, no API key required.

This is a deliberate architectural choice. The system handles potentially sensitive economic analysis. Sending every chunk to an external embedding API introduces latency, cost, and a data exfiltration surface. For a domain-specific retrieval task where the entity extractor and knowledge graph handle the heavy reasoning, embeddings only need to provide reasonable semantic similarity — not world-class performance.

Query embeddings are cached using an OrderedDict-based LRU cache, so repeated queries (common during multi-agent deliberation where several agents ask similar questions) do not regenerate embeddings.

The Lesson

RAG is not just "embed and search." For domains with structured entities and typed relationships — financial analysis, medical records, legal documents, supply chain data — a knowledge graph is not a nice-to-have. It is the difference between finding similar text and actually reasoning about the connections in your data.

The combination of structure-aware chunking, deterministic entity extraction, graph traversal, and principled fusion through RRF produces retrieval quality that pure vector search cannot match for this domain. And it does it without a single external API call.

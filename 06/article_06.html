96 AI Agents, One GPU: Orchestrating a Synthetic Federal Reserve Beige Book

Multi-agent coordination, bounded concurrency against a single GPU, batched vs. granular LLM strategies, and why the first version was catastrophically slow.

Eight times a year, the Federal Reserve publishes the Beige Book — a qualitative survey of economic conditions across all twelve districts. Unlike hard data in jobs reports or CPI releases, the Beige Book trades in anecdote and narrative: what are business contacts actually saying about hiring, spending, and whether the vibes are good or bad?

My platform needs this for every simulation date — including dates where no official Beige Book was published. So I built a service that generates synthetic Beige Book reports using a coordinated team of AI agents. It is 13,678 lines of Python, the most architecturally complex service in the system, and the engineering challenges it presented — multi-agent orchestration, bounded concurrency against a single GPU, graceful degradation at every level — are problems that do not show up in tutorial projects.

Four Agent Types, One District Pipeline

Each of the 12 districts runs a sequential pipeline of four specialized agents:

IntelligenceAgent — The data gatherer. It calls the Economy Service to fetch district-level indicators, recent news, historical minutes, and past Beige Book editions. All date-bounded — agents only see information available on the report date.

SectorAnalysts — A team of eight parallel agents, each specializing in one economic sector: Consumer Spending, Manufacturing, Labor Markets, Real Estate, Banking & Finance, Prices & Inflation, Agriculture & Energy, and Services & Transportation. Each analyst receives the intelligence report and produces a structured analysis with summary, key findings, outlook label, and confidence score.

SynthesizerAgent — Takes the eight sector analyses and weaves them into a coherent district narrative. No tools — works purely from sector outputs. Determines an overall outlook using weighted sector scores.

VerificationAgent — The fact-checker. Cross-references claims in the synthesized report against source documents using TF-IDF cosine similarity. Claims with insufficient source support get flagged.

The temperature tuning reflects each role: intelligence gathering at 0.3 (factual, deterministic), sector analysis at 0.5 (moderate creativity for narrative generation), verification back to 0.3 (rigorous fact-checking).

The Concurrency Challenge

Here is where theory meets hardware. In "granular" mode, each district spawns 8 sector analysts — that is 96 potential concurrent LLM calls, plus intelligence, synthesis, and verification stages. On paper, you want maximum parallelism. In practice, you are hitting a single Ollama instance with limited GPU memory.

The first version used asyncio.gather across all 12 districts. It looked beautiful in the code and was catastrophically slow. Dozens of LLM calls queued behind a global semaphore, and per-district timeouts fired while tasks were still waiting their turn.

The solution was counterintuitive: process districts sequentially.

async def execute_all(self, district_configs):
    """Districts sequentially — each gets exclusive GPU access.
    Avoids the stampede problem where parallel districts queue
    dozens of LLM calls, causing per-district timeouts."""
    final_results = []
    for dc in district_configs:
        workflow = DistrictWorkflow(self.config, dc, ...)
        result = await asyncio.wait_for(
            workflow.execute(),
            timeout=self.config.timeout_per_district,
        )
        final_results.append(result)

Sequential district processing with intra-district parallelism was 3x faster on a single-GPU setup because it eliminated queue contention. Within each district, the sector analysts can run concurrently since the intelligence data is already fetched. The bottleneck is at the LLM call level, not the data preparation level. For reference, the LLM deliberation service (a simpler workload with 22 statements across 5 rounds) takes 9.5 minutes on the same GPU -- so 12 districts with intelligence gathering, sector analysis, synthesis, and verification per district lands in the 10-20 minute range depending on model load.

Batched Mode: The Single Most Impactful Optimization

The "batched" analysis mode covers all sectors in a single LLM call per district — one prompt, one response, all sectors at once. Twelve calls total instead of ninety-six.

BATCHED_SYSTEM_PROMPT = """\
You are writing the Federal Reserve Beige Book.
Write all 8 sector sections in a single response.
Use these exact section headers:

### Consumer Spending
### Manufacturing
### Labor Markets
### Real Estate and Construction
### Banking and Finance
...
"""

The response is parsed back into individual SectorAnalysis objects using regex section splitting:

_HEADING_TO_SECTOR = {
    "consumer spending": "consumer_spending",

If the batched parse returns fewer than 4 sectors — the model sometimes drops sections or produces unparseable output — the service falls back to the granular pipeline for missing sectors. This graceful degradation pattern appears throughout the service.

Pipelined Execution: 1-Ahead Prefetching

Batched mode goes further with a PipelinedExecutor that uses an asyncio.Queue for 1-ahead data prefetching. While district N is running its LLM call, district N+1's intelligence data is being fetched from the Economy Service in parallel:

class PipelinedExecutor:
    async def execute_all(self, district_configs):
        prefetch_queue = asyncio.Queue(maxsize=1)

        async def prefetch_producer():
            for dc in district_configs:
                intel = await _run_intelligence(self.config, dc, ...)
                await prefetch_queue.put((dc, intel, None))
            await prefetch_queue.put(None)  # sentinel

        producer_task = asyncio.create_task(prefetch_producer())

        while True:
            item = await prefetch_queue.get()
            if item is None:
                break
            dc, intelligence, _ = item
            # Run batched analysis, synthesis, verification...

The maxsize=1 is critical — at most one district's intelligence data is buffered ahead, preventing memory bloat when the LLM is slow.

Cross-District Analysis

After all districts complete, the ComparisonTableGenerator produces the national picture: outlook comparison tables (12 districts × 8 sectors), numerical heatmap data, and theme frequency analysis that identifies cross-cutting patterns like "tight labor market" or "wage pressures" appearing across multiple regions.

Background Processing

A full 12-district report takes 10-20 minutes. The API supports background mode — the endpoint returns immediately with a task_id, and clients poll for progress:

@app.get("/analysis")
async def run_analysis(request, date, background=False):
    if background:
        task_id = uuid.uuid4().hex[:12]
        progress.set(task_id, {"status": "running", "pct": 0})
        asyncio.create_task(_run_bg())
        return JSONResponse({"task_id": task_id, "status": "started"})

The on_district_done callback fires after each district, giving the dashboard sub-district progress granularity in real time.

Lessons Learned

Concurrency must match hardware, not theory. Sequential district processing with intra-district parallelism was 3x faster than full parallelism on a single GPU. The correct concurrency model depends on your bottleneck.

LLM calls need graceful degradation at every level. Batched mode falls back to granular if parsing fails. Granular falls back to rule-based output if the LLM is unavailable. The verification agent falls back from TF-IDF to keyword overlap if scikit-learn is missing. Every LLM call has a non-LLM fallback.

Structured output from LLMs is a parsing problem, not a prompting problem. Getting reliable 8-section markdown required robust parsing — regex section splitting, fuzzy heading matching, default values for missing fields — not more elaborate prompts.

Background processing is table stakes for multi-minute operations. The task_id + progress polling pattern is simple to implement and dramatically improves the user experience versus long-running HTTP requests that timeout.

The Beige Book service is the piece of this platform I am most proud of — not because it is flashiest, but because it forced me to solve real distributed systems problems in the context of something genuinely novel.

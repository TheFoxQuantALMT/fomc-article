Three Ways to Build AI Agents (And When to Use Each)

Agent framework patterns for local LLMs — from lightweight tool-callers to full GroupChat orchestration with context injection and graph-based workflows.

When most people hear "AI agent framework," they think of a chatbot with a few tools bolted on. I built something different: a platform where 19 autonomous AI agents role-play as Federal Reserve officials, deliberate monetary policy across structured meeting rounds, and arrive at a consensus rate decision — all running on local LLMs through Ollama.

This article walks through the three distinct agent creation patterns I converged on across five microservices, the GroupChat orchestration that makes multi-agent meetings work, and the WorkflowBuilder that coordinates the entire ten-service pipeline as a directed graph. If you are building multi-agent systems beyond toy demos, these are the patterns that matter.

Running Agent Framework with Local LLMs

A deliberate constraint: all inference runs through Ollama. No OpenAI, no Anthropic API, no cloud inference. The OllamaChatClient makes this possible:

from agent_framework.ollama import OllamaChatClient

chat_client = OllamaChatClient(
    host=args.base_url,
    model_id=args.model,
    keep_alive=300,  # Keep model in VRAM for 5 minutes
)

The keep_alive parameter is critical. When cycling through 19 agents across 5 meeting rounds, you do not want the model unloading and reloading between every turn. The client is shared across all agents in a meeting — Ollama serializes requests to a single GPU anyway, so sharing avoids connection overhead without sacrificing throughput.

Pattern A: Standard ChatAgent (Embarrassingly Parallel)

The belief-formation service creates 19 independent ChatAgent instances, one per FOMC member, and runs them gated by a semaphore for VRAM safety:

from agent_framework import ChatAgent

chat_agent = ChatAgent(
    chat_client=chat_client,
    instructions=build_system_instructions(fomc_agent),
    name=name,
    context_providers=ctx_provider,
    temperature=args.temperature,
)

response = await chat_agent.run(user_message)

Use this when agents are independent and do not need to interact. Each gets persona-specific instructions built from biographical data — tenure, education, policy contributions, communication style.

The Beige Book service uses the same pattern with four specialized agent types — intelligence gatherers (temperature 0.3), sector analysts (0.5), synthesizers (0.5), verifiers (0.3) — each with different tool configurations.

Pattern B: FrameworkAgent with Context Providers and Tools

The deliberation service needs agents that are aware of meeting state, can call tools to look up data, and receive dynamic context that changes between rounds:

from agent_framework import Agent as FrameworkAgent, BaseContextProvider

agent = FrameworkAgent(
    client=self._chat_client,
    name=a.name,
    instructions=_build_system_prompt(a),
    default_options=ChatOptions(temperature=0.7, keep_alive=300),
    context_providers=all_providers,
    tools=agent_tools,
)

The context_providers list includes a DeliberationContextProvider that injects per-agent belief state and — critically — rewrites conversation history. This solves the most subtle bug in multi-agent GroupChat.

The GroupChat Role-Rewriting Bug

When the GroupChat broadcasts Agent A's response, it arrives at Agent B as an assistant-role message. But the LLM interprets assistant messages as its own prior output — it thinks it already said that, and continues the train of thought instead of responding as a different character.

class DeliberationContextProvider(BaseContextProvider):
    async def before_run(self, agent, session, context, state):
        rewritten = []
        for msg in context.input_messages:
            if msg.role == "assistant":
                author = getattr(msg, "author_name", None) or "Another member"
                rewritten.append(Message(
                    role="user",
                    text=f"[{author}]: {msg.text}",
                ))
            else:
                rewritten.append(msg)
        context.input_messages[:] = rewritten

This is invisible in demos but breaks everything in production. Without it, every agent after the first one produces a continuation of the previous agent's response rather than their own perspective.

Pattern C: Lightweight .as_agent() (Utility Agents)

The member-service needs a quick agent for stance analysis — no orchestration, just an LLM client with instructions and tools:

client = OllamaChatClient(host=ollama_host, model_id=model_id)
agent = client.as_agent(
    name="StanceAnalyzer",
    instructions=STANCE_INSTRUCTIONS,
    tools=create_member_tools(session),
)

Use this for utility agents embedded in a service that is not primarily an agent framework application.

GroupChat: Structured FOMC Meetings

The deliberation phase runs up to 19 agents in a structured five-round meeting using GroupChatBuilder with a custom speaker selector:

from agent_framework_orchestrations import GroupChatBuilder

class _RoundSpeakerSelector:
    """Cycles through a fixed speaker list once."""
    def __init__(self, speaker_names):
        self._names = speaker_names
        self._idx = 0

    def __call__(self, state):
        if self._idx >= len(self._names):
            return None
        name = self._names[self._idx]
        self._idx += 1
        return name

group_chat = GroupChatBuilder(
    participants=fw_speakers,
    selection_func=_RoundSpeakerSelector(speaker_names),
    max_rounds=len(speaker_names),
).build()

result = await asyncio.wait_for(
    group_chat.run(prompt),
    timeout=max(600, 120 * len(speaker_names)),
)

The custom selector is important — FOMC meetings are highly structured, not free-form discussions. Different rounds have different speaker pools selected upstream by the FOMCMeetingSpeakerSelector, and the per-round selector simply cycles through the resulting list.

The dynamic timeout (120 × number of speakers) handles the reality that Ollama serializes inference on a single GPU. Eight speakers at ~120 seconds each need a 16-minute timeout.

WorkflowBuilder: Graph-Based Pipeline Management

The full simulation is a multi-phase pipeline modeled as a directed graph with fan-out and fan-in:

from agent_framework import WorkflowBuilder

builder = WorkflowBuilder(start_executor=economy)
builder.add_edge(economy, beige_book)
builder.add_edge(beige_book, belief)
builder.add_fan_out_edges(belief, [mc, llm])
builder.add_fan_in_edges([mc, llm], comparison)
builder.add_edge(comparison, persist)

workflow = builder.build()

After belief formation, the pipeline fans out to run Monte Carlo (CPU-bound) and LLM deliberation (GPU-bound) in parallel, then fans in at comparison to merge results. Each Executor receives state as JSON, does its work, and sends state to the next node — checkpoint-friendly by design.

The deliberation service also uses WorkflowBuilder internally for its meeting phases: workflows within workflows, each level using the same primitives.

Tools: @ai_function for Schema Generation

Agent tools are decorated with @ai_function, which auto-generates JSON schemas from type annotations:

@ai_function(name="rag_search",
             description="Semantic search across FOMC documents.")
async def rag_search(
    query: Annotated[str, "Natural-language search query"],
    n_results: Annotated[int, "Maximum results"] = 5,
    as_of_date: Annotated[str, "ISO date bound"] = "",
) -> str:
    results = await client.context_search(
        query=query, as_of_date=as_of_date, top_k=n_results,
    )
    return json.dumps(results, default=str)

The type annotations do double duty — type checking for the developer, schema generation for the LLM. The as_of_date parameter ties back to the point-in-time constraint from the economy service.

Lessons Learned

Three patterns, not one. ChatAgent for independent parallel work, FrameworkAgent for stateful interactive agents, .as_agent() for lightweight utilities. Knowing which to reach for at which complexity level is the design skill.

Context providers are the most powerful pattern. They fire before each agent turn and can rewrite history, inject state, and personalize the conversation. The role-rewriting fix alone justified the entire framework.

GroupChat needs custom speaker selection. Default free-form selection does not work for structured protocols. Custom selectors give you control over who speaks when.

Pre-release frameworks require tracking breaking changes. I ran versions from 1.0.0b260130 through 1.0.0rc2. Import paths changed, API signatures shifted, and the agent-framework package itself split into sub-packages. This is the cost of working at the frontier.
